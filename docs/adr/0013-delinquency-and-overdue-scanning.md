# 0013. 연체는 상태 있는 상환 회차 + 주기 스캔으로 처리한다

- **Status**: Accepted
- **Date**: 2026-06-20

## Context

지금까지 상환 스케줄은 `Loan`의 원금·이자율·기간으로부터 매번 **도출**(`EqualPaymentScheduleGenerator`)되는
순수 값이었다. 그러나 연체는 "그 회차가 납기까지 납입됐는가"라는 **시간에 따라 변하는 상태**를 요구한다.
도출만으로는 "어느 회차가 아직 안 들어왔는가", "언제부터 연체인가", "연체료가 얼마인가"를 표현할 수 없다.
또한 연체 판정은 특정 시점(납기)을 지나야 발생하므로, 누군가 **주기적으로 깨워서** 판정·기록해야 한다.

## Decision

상환 회차를 **상태 있는 영속 엔티티**로 승격하고, 연체를 **주기 스캔 + 트랜잭셔널 아웃박스**로 처리한다.

- **상태 있는 회차**: `ScheduledRepayment`(`DUE`/`PAID`/`OVERDUE`)를 `loan_repayment` 테이블(Flyway V7)에
  영속화한다. 대출 실행 시(`CreateLoanService`) 회차 N의 납기 = 기준일 + N개월로 스케줄을 함께 적재한다.
- **상태 전이는 도메인 안에**: `markPaid()`(정산 시), `markOverdue(asOf, rate)`(스캔 시)만 상태를 바꾼다.
  두 전이 모두 **멱등** — 이미 PAID/OVERDUE 거나 아직 납기 전이면 아무 일도 하지 않는다.
- **연체료(지연배상금) 공식**: `DelinquencyCalculator` — 순수 도메인.
  `연체료 = 연체원리금 × 연체이자율(연) × 연체일수 / 365`, **원 단위 절사(DOWN)**.
  전 과정 `BigDecimal` 정수 연산으로 1원 오차도 없다. 연체이자율은 대출의 약정 이자율을 사용한다.
- **주기 스캔**: `OverdueScanner`(`@Scheduled`, 기본 60초)가 `findOverdueCandidates`(status=DUE & 납기 경과,
  `(status, due_date)` 인덱스)를 조회한다.
- **회차당 독립 트랜잭션**: 각 후보는 `OverdueProcessingService.process()`에서 독립 트랜잭션으로 처리되어
  한 건 실패가 전체 스캔을 막지 않는다. 상태 변경(OVERDUE + 연체료)과 `RepaymentOverdue` 이벤트 적재를
  **같은 트랜잭션**으로 묶는다(트랜잭셔널 아웃박스, ADR-0009).
- **정산과의 연결**: `SettleRepaymentService`는 원장이 실제 반영된 경우(`applied=true`)에만 해당 회차를
  `markPaid()`로 전이해 연체 대상에서 제외한다.

## Consequences

- **긍정**: 연체 상태·연체료가 영속 사실로 남아 조회(`GET /api/loans/{id}/repayments`)·후속 워크플로의
  근거가 된다. 전이가 멱등이라 스캔 재실행·중복에 안전하고, 회차당 독립 트랜잭션으로 부분 실패가 격리된다.
  연체료는 정수 연산이라 정합성이 보장되고, 순수 도메인 단위 테스트로 빠르게 검증된다.
- **부정/비용**: 상환 스케줄이 이제 **도출값이자 영속 상태**라는 이중성을 가진다(둘이 어긋나지 않게 생성 시
  한 번만 적재). `@Scheduled` 폴링 지연만큼 연체 판정이 늦고, 다중 인스턴스 배포 시 스캔 중복 실행
  방지(샤딩/리더 선출)가 필요하다. 윤년·휴일을 무시한 365 고정 분모는 단순화다.
- **후속**: `RepaymentOverdue` 구독자(연체 알림·추심), 연체 해소(연체 회차의 사후 납입) 처리,
  다중 인스턴스 스캔 조정, 대출 완납/만기 종료 전이.

## Alternatives considered

- **스케줄을 계속 도출만 하고 별도 "납입 기록" 테이블만 둔다** — 상태/연체료를 매 조회 시 재계산해야 하고
  연체료 산정 시점·일수가 모호해진다. 사실(연체료 확정액)을 남기지 못한다.
- **연체 판정을 정산/조회 시점에 lazy 계산** — 아무도 상환하지 않는 대출은 영원히 연체로 전이되지 않아
  연체 이벤트·후속 워크플로를 트리거할 수 없다. 능동적 스캔이 필요하다.
- **DB 스케줄러(pg_cron 등)** — 도메인 로직(연체료·이벤트 적재)이 SQL로 새어 나가고 테스트가 어렵다.
  애플리케이션 레벨 `@Scheduled`로 충분하다.
