# 0002. 멀티모듈 모듈러 모놀리스 + 클린 아키텍처

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

대출·투자·상환·정산·원장은 서로 다른 비즈니스 컨텍스트다. 결합도가 높으면 한 컨텍스트의 변경이
다른 곳으로 번지고, 도메인 로직이 프레임워크에 묶이면 테스트와 유지보수가 어려워진다.
"낮은 결합도·높은 응집도, 테스트 가능한 구조"가 핵심 요구사항이다.

## Decision

Gradle **멀티모듈**로 컨텍스트별 모듈(`ledger`, `investment`, `lending`, …)을 나누고,
각 모듈 내부를 **클린 아키텍처 계층**(domain → application → adapter)으로 구성한다.

- `domain`: 순수 코틀린. 프레임워크 의존 0 (Spring/JPA import 금지).
- `application`: 유스케이스 + 포트(인터페이스). 트랜잭션 경계.
- `adapter`: JPA/REST 등 인프라 구현. 의존 방향은 항상 `adapter → application → domain`.
- `common`: Money 등 금융 원시 타입. `bootstrap`: Spring Boot 조립(유일하게 boot 플러그인 적용).

처음부터 분산 마이크로서비스로 가지 않고 **모듈러 모놀리스**로 시작해, 이후 모듈을 서비스로 추출 가능하게 둔다.

## Consequences

- **긍정**: 의존성 방향이 빌드 파일로 강제된다(`common`엔 Spring 의존성이 한 줄도 없음 = 도메인 순수성의 증명).
  도메인을 인프라 없이 단위 테스트할 수 있다.
- **부정/비용**: 모듈/계층이 늘어 초기 보일러플레이트가 많다. 모듈 간 통신 규약을 따로 정의해야 한다.
- **후속**: 계층 의존 규칙을 ArchUnit 테스트로 강제(예정). 컨텍스트 간 통신은 도메인 이벤트로(EDA, 예정).

## Alternatives considered

- **단일 모듈 패키지 분리** — 빠르지만 계층/컨텍스트 경계가 컴파일 타임에 강제되지 않음.
- **처음부터 마이크로서비스** — 1인 포트폴리오 규모에 운영 복잡도가 과함. 모듈러 모놀리스로 시작.
