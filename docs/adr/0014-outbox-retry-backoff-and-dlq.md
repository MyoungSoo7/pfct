# 0014. 아웃박스 발행 실패는 백오프 재시도 후 DLQ로 격리한다

- **Status**: Accepted
- **Date**: 2026-06-20

## Context

트랜잭셔널 아웃박스(ADR-0009)의 릴레이는 미발행 이벤트를 폴링해 발행하고, 실패하면 `published_at` 을
찍지 않아 다음 폴링에서 재시도했다(at-least-once). 그러나 기존 릴레이는 배치 전체를 하나의 트랜잭션에서
처리했기 때문에 두 가지 문제가 있었다.

- **머리 막힘(head-of-line blocking)**: 배치 중 한 건이라도 발행에 실패하면 트랜잭션이 롤백되어,
  이미 성공적으로 Kafka 로 보낸 건의 `published_at` 도 함께 사라졌다(다음 폴링에서 중복 재발행).
- **독약 메시지(poison message)**: 영원히 실패하는 이벤트(직렬화 깨짐, 잘못된 토픽 등)가 폴링 맨 앞에
  남아 뒤 이벤트의 발행을 무한정 막을 수 있었다. 재시도 간격도 폴링 주기에 고정되어 백오프가 없었다.

분산 환경에서 브로커 일시 장애는 정상 상황이므로, "유한 횟수까지는 점점 뜸하게 재시도하고, 그래도 안 되면
정상 흐름에서 떼어내 격리"하는 표준 운영 모델이 필요하다.

## Decision

릴레이를 **이벤트 단위 처리 + 지수 백오프 재시도 + DLQ 격리**로 바꾼다.

- **이벤트 단위 결과 처리**: 한 배치에서 건마다 `try/catch` 한다. 성공은 즉시 `published_at` 을 찍고,
  실패는 그 건만 재시도 상태로 바꾼다 — 한 건의 실패가 같은 배치의 다른 건을 되돌리지 않는다.
- **지수 백오프**: 실패 시 `attempts` 를 늘리고 `next_attempt_at = now + base × 2^(attempts-1)`(상한 존재)로
  다음 시도를 미룬다. 발행 후보 조회는 `published_at is null and dead = false and
  (next_attempt_at is null or next_attempt_at <= now)` 로, 대기 중인 건을 건너뛴다.
- **DLQ 격리**: `attempts` 가 한도(`pfct.outbox.max-attempts`, 기본 5)에 도달하면 `dead = true` 로 표시해
  더는 폴링되지 않게 하고, `DeadLetterPublisher` 로 데드레터 토픽(`pfct.outbox.dlq`)에 흘려보낸다.
  **DB 의 `dead` 플래그가 권위 있는 격리 기록**이고 Kafka DLQ 전송은 부가(알림/재처리용)다 —
  DLQ 전송이 실패해도 `dead` 는 유지된다.
- **운영 가시성**: 격리 사유는 `last_error` 에 남기고, `GET /api/admin/outbox/dead` 로 조회한다.
- **결정성 있는 테스트**: 릴레이는 `publishReadyAt(now)` 에 시각을 주입받아, 실시간/스케줄에 의존하지 않고
  백오프 경계와 격리를 검증한다(`OutboxDlqIntegrationTest`).
- 스키마는 Flyway `V8` 로 `attempts / next_attempt_at / last_error / dead` 컬럼과 발행 후보·DLQ 부분
  인덱스를 추가한다(JPA 는 `validate`).

## Consequences

- **긍정**: 일시 장애에 백오프로 탄력 있게 대응하고, 독약 메시지가 파이프라인을 막지 못한다.
  성공 건이 실패 건과 함께 롤백되지 않아 불필요한 중복 재발행이 줄고, 격리·사유·시도 횟수가 영속 사실로
  남아 운영 진단과 재처리가 가능하다. 여전히 at-least-once(중복 가능)이며 소비측 멱등(ADR-0012)이 이를 받친다.
- **부정/비용**: 자동 재처리(DLQ → 정상 토픽 되돌리기)는 아직 수동 영역이다. 백오프는 단일 릴레이 인스턴스의
  벽시계에 기반하므로 정밀한 분산 스케줄링은 아니다. `next_attempt_at` 조건이 더해져 후보 조회가 약간 복잡해졌다.
- **후속**: DLQ 재처리(되돌리기) 엔드포인트, 멀티 인스턴스 릴레이에서의 폴링 경합 제거(ADR-0015),
  발행 지표(성공/재시도/격리 카운트) 노출.

## Alternatives considered

- **무한 재시도(현행 유지)** — 독약 메시지가 폴링을 영구히 막고, 배치 전체 롤백으로 성공 건까지 재발행된다.
- **즉시 DLQ(재시도 0회)** — 일시적 브로커 장애에도 곧장 격리되어 운영 부담이 커진다. 대부분의 실패는 일시적이다.
- **Kafka DLQ만 두고 DB 에는 상태를 남기지 않음** — 격리 사실/사유가 휘발되어 조회·재처리 근거가 사라지고,
  DLQ 전송 실패 시 이벤트가 영구 유실될 수 있다. DB 를 권위 기록으로 삼는다.
- **Spring Kafka 의 `DefaultErrorHandler`/`DeadLetterPublishingRecoverer`(소비측 재시도)** — 그것은
  *컨슈머* 의 처리 실패를 다룬다. 여기 문제는 *아웃박스 릴레이(프로듀서측)* 의 발행 실패라 적용 지점이 다르다.
  (소비측 DLQ 는 향후 별도로 둘 수 있다.)
