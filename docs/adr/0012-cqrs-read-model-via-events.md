# 0012. 투자자 수익 조회는 이벤트 기반 CQRS 읽기 모델로

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

투자자 대시보드(누적 수익 등)는 조회가 잦고 화면 친화적인 형태가 필요하다. 이를 정규화된 쓰기 모델
(원장 분개)에서 매번 집계 쿼리로 만들면 비용이 크고 도메인 모델이 조회 관심사에 오염된다.

## Decision

**CQRS** 로 읽기 모델을 쓰기 모델과 분리한다.

- **쓰기측**: 정산이 반영되면 `RepaymentSettled` 이벤트를 아웃박스로 적재(원장 기록과 같은 트랜잭션).
- **전송**: 아웃박스 릴레이 → Kafka(ADR-0009/0011).
- **읽기측**: `InvestorReturnsProjector`(@KafkaListener)가 이벤트를 구독해 비정규화 읽기 모델
  `investor_return_view`(투자자별 누적 수익)를 갱신한다.
- **읽기측 멱등**: Kafka 는 at-least-once 이므로 정산 ID를 `processed_event` 에 기록해 중복 반영을 막는다.
- 조회는 읽기 모델만 본다(`GET /api/investors/{id}/returns`).

## Consequences

- **긍정**: 조회가 단일 테이블 lookup 으로 빨라지고, 쓰기 도메인이 조회 관심사로부터 자유로워진다.
  읽기 모델은 이벤트로 독립적으로 진화/재구축 가능. 실제 Kafka 통합 테스트로 투영·멱등 검증.
- **부정/비용**: 읽기 모델은 **최종적 일관성**(이벤트 처리까지 약간의 지연). 모델이 늘면 프로젝터/스키마 관리 필요.
- **후속**: 읽기 모델 재구축(리플레이) 절차, 추가 뷰(포트폴리오/연체 현황 등), 스키마 버저닝.

## Alternatives considered

- **쓰기 모델에 집계 쿼리** — 단순하지만 조회 비용·도메인 오염 문제. 트래픽이 늘면 한계.
- **DB 트리거/뷰로 동기 갱신** — 강결합이고 이벤트 기반 확장성이 없다.
