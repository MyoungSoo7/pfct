# 0010. 상환금 정산은 최대 잉여 비율 분배 + 원장 기록

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

차주가 납입한 상환금(원리금)을 투자자에게 **투자 비율대로** 돌려줘야 한다. 비율로 나누면 단수(端數)가
생기는데, 각자 반올림하면 분배 합이 원금과 1~N원 어긋나 정합성이 깨진다. 또 플랫폼은 이자에 대한
수수료를 가져가야 하고, 모든 자금 이동은 원장에 균형 있게 남아야 한다.

정산 비율을 계산하려면 투자자별 기여액이 필요하므로, 집계만 저장하던 것을 **개별 투자 내역**으로도 저장한다.

## Decision

- **개별 투자 내역 저장**: `investment` 테이블에 (round, investor, amount)를 append-only로 기록(투자 시 같은 tx).
- **비율 분배는 최대 잉여(largest remainder) 방식**: `ProRataDistributor`(순수 도메인, `modules/settlement`).
  각 몫을 내림한 뒤, 남은 금액을 잉여가 큰 순서로 1원씩 배분 → **분배 합 == 분배 대상 금액**(1원도 안 샘).
- **수수료**: 이자에 수수료율을 적용해 원 단위 절사, 플랫폼이 가져간다.
- **원장 기록**(차변=대변): 대변 borrower(원리금), 차변 investor[i](비율 분배액), 차변 platform:fee(수수료).
- **멱등**: 거래 ID `settle:{loanId}:{seq}` 로 같은 회차 중복 정산을 차단.

## Consequences

- **긍정**: 분배가 1원도 어긋나지 않고 원장이 항상 균형을 이룬다(돈 보존). 분배기는 순수 함수라 단위 테스트로
  완전 검증(6 케이스). 실제 Postgres 통합 테스트로 비율 분배·수수료·멱등 검증.
- **부정/비용**: 투자 내역을 행 단위로 저장하므로 데이터가 늘어난다. 분배 대상이 매우 많으면 정렬 비용 발생.
- **후속**: 회차별 상환 스케줄 추적/상태 관리, 정산 결과를 읽기 모델로 투영(CQRS, Phase C 후속).

## Alternatives considered

- **각자 비율로 나눠 단순 반올림** — 구현은 쉽지만 합계가 원금과 어긋나 금융에서 부적합.
- **마지막 투자자에게 단수 몰아주기** — 합계는 맞지만 특정 투자자에게 불리/유리하게 치우침.
  잉여가 큰 쪽에 주는 최대 잉여 방식이 더 공정.
