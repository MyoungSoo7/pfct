# 0007. 멱등한 append-only 원장 기록

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

분산/이벤트 기반 환경에서는 같은 명령이 두 번 전달될 수 있다(at-least-once 메시징, 클라이언트 재시도,
Saga 재실행). 원장에 같은 자금 이동이 중복 기록되면 잔액이 틀어진다. 또한 원장은 감사 가능해야 하므로
기록을 사후 수정/삭제하면 안 된다.

## Decision

원장을 **append-only**로 영속화한다(`ledger_transaction` 헤더 + `journal_entry` 라인, 수정/삭제 없음).
잔액은 분개의 합 `Σ(차변) − Σ(대변)`으로 도출한다.

**멱등성**은 거래 ID를 멱등 키로 삼아 보장한다.

- 호출자가 자금 이동의 **자연 키**로 거래 ID를 부여한다(예: `disburse:{loanId}`).
- 기록 전 존재 여부 확인(check-first) → 이미 있으면 `applied=false`로 건너뛴다.
- 최종 방어선은 `ledger_transaction.id` **PK 유니크 제약**이다.

## Consequences

- **긍정**: 중복 요청이 와도 잔액이 한 번만 반영된다. 원장이 그대로 감사 로그가 된다.
  실제 PostgreSQL 통합 테스트로 잔액 도출·멱등성 검증 완료.
- **부정/비용**: 잔액을 매번 합산하면 계정 분개가 많아질수록 비용 증가 → 추후 잔액 스냅샷/마감 도입 여지.
  check-first + PK는 드문 동시 중복(같은 ID 병렬) 시 한쪽이 제약 위반으로 롤백되지만, 동일 논리 연산이라 무방.
- **후속**: Saga의 각 단계가 이 멱등 기록을 사용해 안전하게 재시도 가능(ADR 예정: Saga/Outbox).

## Alternatives considered

- **별도 멱등 키 테이블** — 범용적이지만, 원장은 거래 ID 자체가 자연 멱등 키라 PK로 충분.
- **가변 잔액 컬럼 직접 갱신** — 빠르지만 append-only 감사 추적성과 중복 방어를 잃음.
