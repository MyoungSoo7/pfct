# 0015. 아웃박스 릴레이는 FOR UPDATE SKIP LOCKED로 멀티 인스턴스 안전하게 만든다

- **Status**: Accepted
- **Date**: 2026-06-20

## Context

아웃박스 릴레이(ADR-0009, ADR-0014)는 발행 후보를 `published_at is null` 등의 조건으로 조회해
발행하고 상태를 갱신한다. 지금까지의 조회는 일반 `SELECT` 였고, 암묵적으로 **릴레이 인스턴스가 하나**라고
가정했다.

그러나 가용성·처리량을 위해 애플리케이션은 보통 2대 이상으로 배포된다. 각 인스턴스의 `@Scheduled` 릴레이가
같은 후보 행을 동시에 조회하면 **같은 이벤트를 둘 다 발행**한다. at-least-once + 소비측 멱등(ADR-0012)으로
정합성은 깨지지 않지만, 불필요한 중복 트래픽과 경합이 발생하고 "수평 확장 가능"하다고 말할 수 없다.

필요한 것은, 여러 릴레이가 동시에 폴링해도 **서로 겹치지 않는 후보 집합**만 가져가게 하는 것이다.

## Decision

발행 후보 조회(`findReadyForPublish`)에 **행 잠금 + `SKIP LOCKED`** 를 적용한다.

- JPA `@Lock(PESSIMISTIC_WRITE)` + 잠금 타임아웃 힌트 `jakarta.persistence.lock.timeout = -2`
  (= Hibernate `LockOptions.SKIP_LOCKED`) → PostgreSQL `SELECT … ORDER BY id … LIMIT … FOR UPDATE SKIP LOCKED`.
- 한 릴레이의 트랜잭션이 선택한 행에 행 잠금을 걸고, **다른 트랜잭션이 이미 잠근 행은 결과에서 건너뛴다**.
  덕분에 인스턴스마다 disjoint 한 배치를 받아 중복 발행 없이 병렬로 진행한다.
- **트랜잭션 경계가 핵심**: `select → 발행 → published_at 갱신`이 릴레이 `publishReadyAt` 의 한
  트랜잭션(`@Transactional`) 안에서 일어나야, 발행이 끝나 커밋될 때까지 잠금이 유지되어 다른 인스턴스가
  그 행을 건드리지 못한다. 커밋과 동시에 잠금이 풀리고 행은 `published_at` 이 찍혀 더는 후보가 아니다.
- 발행이 실패해 트랜잭션이 롤백되면 잠금도 풀리고 `attempts/next_attempt_at`(ADR-0014) 갱신만 남아,
  백오프 후 어느 인스턴스든 다시 집어갈 수 있다.

검증: `OutboxRelayConcurrencyTest` — 워커 6개가 같은 아웃박스를 동시 폴링하고 발행기는 일부러 sleep 해
잠금 보유 구간을 넓힌다. 그럼에도 각 이벤트는 **정확히 한 번** 발행된다(중복 0, 누락 0).

## Consequences

- **긍정**: 릴레이를 N대로 띄워도 안전하다 — 폴링 경합/중복 발행이 사라지고 처리량이 인스턴스 수에 비례해
  늘어난다. 리더 선출·샤딩 같은 추가 조정 장치 없이 DB 한 줄짜리 메커니즘으로 해결한다. 잠긴 행을 기다리지
  않고 건너뛰므로(NOWAIT 와 달리) 워커가 블로킹되지 않는다.
- **부정/비용**: PostgreSQL 전용 기능에 의존한다(다른 RDBMS 는 지원 여부/문법이 다를 수 있다). 잠금은
  트랜잭션 경계를 정확히 요구하므로, 릴레이를 트랜잭션 밖에서 호출하면 잠금이 곧바로 풀려 효과가 없다.
  배치 크기가 너무 크면 한 인스턴스가 많은 행을 오래 잠가 다른 인스턴스의 몫이 줄 수 있다(적정 `batch-size`).
- **후속**: 같은 패턴을 연체 스캐너(`OverdueScanner`, ADR-0013의 후속 과제였던 "다중 인스턴스 스캔 조정")에도
  적용 가능. 발행 지표(인스턴스별 처리량) 노출.

## Alternatives considered

- **리더 선출(ShedLock/단일 활성 인스턴스)** — 한 번에 한 인스턴스만 릴레이를 돌린다. 단순하지만
  처리량이 인스턴스 1대로 묶이고, 별도 잠금 저장소/라이브러리 의존이 생긴다. SKIP LOCKED 는 모든 인스턴스가
  동시에 일하면서도 안전하다.
- **파티션/샤딩(인스턴스별 id 범위 분담)** — 정적 분할은 인스턴스 수 변동·불균형에 취약하고 운영이 번거롭다.
- **`FOR UPDATE`(SKIP 없이) 또는 `NOWAIT`** — 전자는 다른 인스턴스가 잠긴 행을 기다리며 블로킹되고,
  후자는 즉시 에러를 던져 재시도 로직이 필요하다. 큐 소비에는 SKIP LOCKED 가 표준이다.
- **조건부 UPDATE 로 선점(낙관적 클레임)** — `update … set claimed_by=? where id in (select … limit n)`
  방식. 가능하지만 별도 클레임 컬럼·만료 처리(죽은 인스턴스가 잡은 행 회수)가 필요해 더 복잡하다.
