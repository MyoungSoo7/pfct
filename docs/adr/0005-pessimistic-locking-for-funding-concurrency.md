# 0005. 펀딩 동시성에 비관적 락 사용

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

모집 한도가 거의 찬 펀딩 라운드에 다수의 투자자가 "막차" 투자를 동시에 시도하면,
경쟁 조건(race condition)으로 목표 금액을 초과(오버펀딩)할 수 있다. 충돌 빈도가 높은 핫 로우(hot row)다.

## Decision

투자 약정 시 `SELECT … FOR UPDATE`(JPA `@Lock(PESSIMISTIC_WRITE)`)로 해당 라운드 행을 잠근다.
트랜잭션 경계 안에서 락을 보유한 채 도메인 불변식(잔여 한도 검증)을 실행하고 커밋 시 해제한다.
→ 같은 라운드에 대한 투자가 행 단위로 직렬화되어 오버펀딩이 구조적으로 불가능해진다.

검증: `FundingConcurrencyTest`가 실제 PostgreSQL에서 200스레드 동시 투자 → 정확히 100건 성공,
`raised == target`(초과 0)을 보장함을 확인했다.

## Consequences

- **긍정**: 정합성이 가장 단순하고 확실하게 보장된다. 충돌이 잦은 시나리오에서 재시도 폭주가 없다.
- **부정/비용**: 같은 라운드의 투자는 직렬화되어 처리량이 제한된다. 락 대기/타임아웃 관리가 필요하다.
- **후속**: 라운드 간에는 병렬 처리되므로 핫 라운드 단위 처리량만 영향. 필요 시 샤딩/예약 패턴 검토.

## Alternatives considered

- **낙관적 락(@Version + 재시도)** — 충돌이 드문 경우 유리하나, 막차 시나리오는 충돌이 잦아 재시도가 폭증.
  대안으로 문서화만 하고 채택하지 않음.
- **DB CHECK 제약에만 의존** — 초과 INSERT/UPDATE는 막지만, 실패를 애플리케이션이 우아하게 다루기 어려움.
  심층 방어로 CHECK는 함께 두되(ADR-0003), 1차 방어는 비관적 락으로 한다.
