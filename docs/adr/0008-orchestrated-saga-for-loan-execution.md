# 0008. 대출 실행은 오케스트레이션 Saga + 보상 트랜잭션으로

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

대출 실행은 여러 컨텍스트에 걸친다: 투자(라운드 실행 확정) → 여신(대출 생성) → 원장(차주 지급).
지금은 단일 DB 모듈러 모놀리스라 한 트랜잭션으로 묶을 수도 있지만, 향후 컨텍스트가 별도 서비스로
분리되면 단일 ACID 트랜잭션이 불가능해진다. 분리 이후에도 그대로 동작할 구조가 필요하다.

## Decision

**오케스트레이션 기반 Saga**를 도입한다. 오케스트레이터(`LoanExecutionSaga`)는 컴포지션 루트(bootstrap)에 두고,
각 단계를 **독립 트랜잭션**(각 컨텍스트의 `@Transactional` 서비스)으로 실행한다.
후속 단계가 실패하면 앞 단계를 **역순 보상 트랜잭션**으로 되돌린다.

- 1단계 라운드 실행 확정(`execute`) ↔ 보상 `compensate`(EXECUTED→FULFILLED)
- 2단계 대출 생성(`create`, 멱등) ↔ 보상 `cancel`(대출 CANCELLED)
- 3단계 지급(원장 기록 + 아웃박스, 멱등)
- 오케스트레이터 자신은 트랜잭션을 열지 않는다. 각 단계가 스스로 커밋한다.

## Consequences

- **긍정**: 컨텍스트가 서비스로 분리돼도 패턴 그대로 유효. 부분 실패가 보상으로 일관성 있게 정리된다.
  실제 Postgres 통합 테스트로 정상/보상/멱등 재실행 3가지를 모두 검증.
- **부정/비용**: 단일 트랜잭션보다 코드가 복잡하고, 각 단계가 멱등·보상 가능해야 한다.
  Saga 진행 중 중간 상태(EXECUTED)가 잠시 외부에 노출될 수 있다(최종적 일관성).
- **후속**: 단계 간 통신을 동기 호출 → 아웃박스/이벤트 기반으로 확장(ADR-0009), 실패·재시도 정책 정교화.

## Alternatives considered

- **단일 ACID 트랜잭션** — 지금은 가장 간단하나, 서비스 분리 시 폐기해야 하므로 학습/시연 가치가 낮음.
- **코레오그래피(이벤트만으로 각자 반응)** — 결합도는 낮지만 전체 흐름 추적/보상이 어렵다.
  복잡한 금융 트랜잭션은 흐름이 명시적인 오케스트레이션이 운영·디버깅에 유리.
