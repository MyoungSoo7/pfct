# 0001. ADR로 설계 결정을 기록한다

- **Status**: Accepted
- **Date**: 2026-06-18

## Context

이 프로젝트는 금융 도메인의 복잡한 설계 결정(정합성 전략, 락 방식, 모듈 경계 등)을 다수 내린다.
결정의 *결과*(코드)만 남으면, 나중에 "왜 이렇게 했는지"의 맥락이 사라진다. 특히 포트폴리오로서
면접에서 설계 의도를 설명해야 하므로, 결정의 근거를 추적 가능한 형태로 남길 필요가 있다.

## Decision

중요한 설계 결정을 `docs/adr/` 아래 Nygard 형식의 ADR 문서로 기록한다.
번호는 단조 증가하며, 한 번 Accepted된 ADR은 수정하지 않고 새 ADR로 대체(Supersede)한다.

## Consequences

- **긍정**: 의사결정의 맥락이 코드와 함께 버전 관리된다. 신규 합류자/리뷰어가 "왜"를 빠르게 파악한다.
- **부정/비용**: 결정을 내릴 때마다 문서 작성 오버헤드가 든다.
- **후속**: 결정 기록 시 `MEMORY.md` / `STATUS.md`를 함께 갱신하는 프로토콜을 `CLAUDE.md`에 둔다.

## Alternatives considered

- **위키/노션 등 외부 문서** — 코드와 분리되어 동기화가 깨지기 쉬움. 레포 내 문서를 택함.
- **커밋 메시지에만 의존** — 흩어져 있어 "현재 유효한 결정"을 한눈에 보기 어려움.
